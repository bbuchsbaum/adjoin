## Summary

This phase focuses on cleaning up the neighborweights package API surface by removing deprecated function aliases and resolving R CMD check issues. The research reveals that the package has already properly deprecated three functions using R's `.Deprecated()` mechanism, and they are ready for complete removal:

- `label_matrix` (deprecated alias for `diagonal_label_matrix_na`)
- `label_matrix2` (deprecated alias for `diagonal_label_matrix`)
- `discriminating_simililarity` (typo version of `discriminating_similarity`)

The internal codebase contains two instances where deprecated aliases are still used (`label_matrix` in knn_weights.R and `label_matrix2` in class_graph.R), which must be updated before removal. The .Rbuildignore issue is straightforward to address, and compiler warnings need investigation focusing on header inclusion order patterns.

**Primary recommendation:** Complete function removal is safe and appropriate. All three functions already use `.Deprecated()`, making this a final-stage cleanup rather than initial deprecation.

### Pattern 1: Safe Function Removal Workflow
**What:** Multi-step process to remove deprecated functions without breaking builds
**When to use:** Any time removing exported functions from an R package
**Steps:**
1. Identify all internal uses of deprecated functions
2. Update internal code to use replacement functions
3. Remove test cases for deprecated functions
4. Remove roxygen2 `@export` and `@rdname` tags
5. Remove function definitions from source files
6. Run `devtools::document()` to regenerate NAMESPACE
7. Run `devtools::check()` to verify no breakage

### Pattern 2: .Rbuildignore Pattern Management
**What:** Regex-based file exclusion from package builds
**When to use:** Excluding development files from built package tarball
**Syntax:**
```
# .Rbuildignore uses Perl regex, one per line, case-insensitive
^\.planning$          # Exact match (anchored)
^\.planning/          # Directory and contents (simpler, recommended)
```

**Best practice:**
- Use `^` anchor for package root-relative paths
- Use `/` suffix for directories to be more explicit
- Use `usethis::use_build_ignore(".planning")` for automatic escaping
- Verify with `devtools::check()` (respects .Rbuildignore) vs `R CMD check .` (does not)

### Pattern 3: Internal Function Reference Updates
**What:** Replace deprecated function calls with current function names
**Example:**
```r
# BEFORE (in R/class_graph.R lines 369-370):

# AFTER:
```

### Anti-Patterns to Avoid
- **Removing functions without updating internal uses:** Causes immediate package check failures
- **Manual NAMESPACE editing:** roxygen2 will overwrite changes; use `@export` tags instead
- **Forgetting to document():** NAMESPACE won't update, leaving broken exports
- **Testing on source with `R CMD check .`:** Doesn't respect .Rbuildignore; use `devtools::check()`

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| .Rbuildignore regex escaping | Manual regex escaping | `usethis::use_build_ignore()` | Automatically escapes special characters, prevents silent exclusion bugs |
| NAMESPACE management | Manual editing | roxygen2 `@export` tags | Roxygen2 overwrites manual edits; maintaining sync is error-prone |
| Finding deprecated usage | Manual grep | `devtools::check()` with `--as-cran` | R CMD check detects usage of `.Deprecated()` functions |
| Deprecation messaging | Custom warnings | Base R `.Deprecated()` | Standard mechanism, automatically formats messages, widely understood |

**Key insight:** R package tooling has mature, well-tested solutions for common maintenance tasks. The devtools/roxygen2/usethis ecosystem handles edge cases (like regex escaping, cross-references, NAMESPACE consistency) that manual approaches often miss.

## Common Pitfalls

### Pitfall 1: Removing @rdname Aliases Without Removing Function Definitions
**What goes wrong:** Documentation builds but package contains zombie deprecated functions that still work
**Why it happens:** Developer removes `@export` tag but leaves function body in source
**How to avoid:**
- Create checklist of all locations for each deprecated function (definition, @export, @rdname, tests)
- Verify removal with `grep -r "function_name" R/` after edits
**Warning signs:** `R CMD check` passes but deprecated function still callable

### Pitfall 2: .Rbuildignore Pattern Too Broad
**What goes wrong:** Pattern like `planning` (unanchored) matches any file containing "planning" anywhere in path
**Why it happens:** .Rbuildignore uses regex matching, not glob patterns
**How to avoid:**
- Always anchor patterns with `^` for package root
- Test pattern with `grepl("^\\.planning", ".planning", perl=TRUE)` in R
- Use `usethis::use_build_ignore()` which escapes dots and anchors automatically
**Warning signs:** `R CMD build` silently removes unintended files; package mysteriously incomplete

### Pitfall 3: R CMD check vs devtools::check() .Rbuildignore Confusion
**What goes wrong:** .Rbuildignore entry exists but `R CMD check .` still reports NOTE about hidden directory
**Why it happens:** `R CMD check` runs on source and doesn't respect .Rbuildignore; it's called "build ignore" not "check ignore"
**How to avoid:**
- Use `devtools::check()` which builds first, then checks the tarball
- Understand that .Rbuildignore only affects `R CMD build` output
- For clean checks, either use `devtools::check()` or run `R CMD build` then `R CMD check tarball.tar.gz`
**Warning signs:** Pattern in .Rbuildignore looks correct but NOTE persists when running `R CMD check .`

**Source:** [R-hub blog on .Rbuildignore](https://blog.r-hub.io/2020/05/20/rbuildignore/)

### Pitfall 4: Updating Internal Code After Removal Instead of Before
**What goes wrong:** Package no longer builds after deprecated function deletion
**Why it happens:** Developer removes function definition before updating internal callers
**How to avoid:**
- Search entire codebase for deprecated function usage FIRST: `grep -r "label_matrix(" R/`
- Update all internal uses to replacement functions
- Verify package still works: `devtools::test()` and `devtools::check()`
- THEN remove deprecated function definitions
**Warning signs:** Test failures or R CMD check errors about undefined functions

### Pitfall 5: Compiler Header Inclusion Order (R 4.5.0+)
**What goes wrong:** Compile-time warnings or errors about R headers included before Rcpp headers
**Why it happens:** R 4.5.0 made `R_NO_REMAP` default for C++ code; Rcpp must set preprocessor flags before R headers load
**How to avoid:**
- Always include Rcpp/RcppArmadillo headers FIRST in .cpp files
- Never directly include R.h, Rinternals.h, or Rdefines.h in Rcpp code
- Let Rcpp manage R header inclusion internally
**Warning signs:** Compiler messages like "R.h has been included before any Rcpp headers"
**Current status:** This package's C++ code correctly includes RcppArmadillo.h first (verified in src/weight_funs.cpp)

**Source:** [Rcpp Issue #1410](https://github.com/RcppCore/Rcpp/issues/1410)

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Manual .Rd file editing | roxygen2 documentation | 2015+ (roxygen2 maturity) | NAMESPACE/documentation sync automated |
| Custom deprecation warnings | `.Deprecated()` base R | Always standard | Standardized user-facing messages |
| `.Defunct()` errors | `.Deprecated()` → removal | R 3.0+ best practice | Modern packages skip .Defunct() stage if already deprecated |
| `R CMD check .` for validation | `devtools::check()` | 2015+ (devtools maturity) | Respects .Rbuildignore, matches CRAN build process |
| Manual escaping in .Rbuildignore | `usethis::use_build_ignore()` | 2018+ (usethis release) | Prevents regex escaping errors |
| C++11 standard | C++14+ for RcppArmadillo | 2025 (RcppArmadillo update) | Modern Armadillo features require C++14 minimum |

**Deprecated/outdated:**
- `.Defunct()` intermediate stage: Modern practice is `.Deprecated()` for one release cycle, then complete removal
- `@S3method` roxygen2 tag: Deprecated in roxygen2 4.0.0, replaced by automatic S3 method detection
- Direct R header inclusion in Rcpp code: R 4.5.0 makes R_NO_REMAP default, Rcpp should manage header order
